# Storage Decisions

Ruling seats for the Storage Architecture Spec and blueprint. Each numbered line is a seat to rule under max purity. The bullet under it names the hardest precise *how* that seat is still swerving from — the choice that, once made, forbids the other hows. Docs and code are evidence to interrogate, not authorities to pattern-match.

Status: seats open until ruled into the Spec. Apps on Kyzo (KyzoMem / KyzoKnow) are out of scope.

**Key**

- **Q:** Question — the hardest precise *how* this seat is still swerving from
- **A:** Answer — the max-purity pick (what is chosen; what becomes unrepresentable)
- **E:** Explanation — why that pick, and what Delete / Refuse / Proof close it

**Claim strength (mandatory Spec tags — every “unrepresentable” re-tags under one of these):**

- **Unconstructible** — no path through the **sealed Rust type and crate boundary** constructs the illegal state. Bindings marshal and never decide. Crash-safe under the sealed program. **Boundary:** a hostile binding, compromised runtime, crafted deserialization, or hostile adapter is **not** in scope for this tag—that is **Unexposed** with named detection. Inflating Unexposed to Unconstructible is Spec fraud. Unconstructible is also relative to **facts the process has observed** (e.g. a recovery root already in the open Store); it never claims to prevent offline dual-history a partitioned process cannot see.
- **Refused** — reachable program; typed runtime outcome.
- **Unexposed** — absent from the public API but reachable by a hostile adapter (cloned disk, snapshot fork, bit-copied keystore, partitioned oblivious writer, hostile binding). Must name the residual risk and detection—never inflate to Unconstructible.

**Carriage vs authority (preamble filter for every external system):** external systems may **execute** Kyzo’s decisions and **accelerate** Kyzo’s checks, but the moment their timing, their probabilistic state, or their cleanup behavior becomes the authority for what exists, carriage has begun interpreting—and that constructor must be pulled back inside.

**Observation vs type boundary (preamble):** epoch advance and fork provide **forward local fencing** for nodes that have observed the successor fact—never prevention of offline dual-history. Convergence is detection plus remediation (dual-chain poison), never conflict-free merge. The same physics governs ForkGrant’s oblivious original and RecoveryGrant’s partitioned holder.

Layer order (bottom → top): identity → truth → seams → durability → time → consistency → local adapter → integrity → confidentiality → federation → hosts → operate & prove → refuse → ship.

### L1 — Identity
1. **Engine vs Store vs Catalog — type-law composition only**
   - **Q:** Which type boundary makes Engine ≠ Store ≠ Catalog unconstructible to reverse — what does Catalog own versus interpret — and which public constructor that collapses them into “the database” is deleted?
   - **A:** Three sealed, non-interchangeable types with no public common super-type: `Store` (sole durable ordered-byte + fsync/`Committed` door; sole owner of bytes, counters, and write lineage), `Catalog` (sole **interpretive** capability: as-of schema evaluation and admission-gating of schema-mutating Records — **owns no bytes, no fsync path, no counters**), `Engine` (sole Record admission + evaluation + projection orchestration — holds `Store`/`Catalog` capabilities by composition, never owns their doors). Schema mutations are KyzoRecords committed through the Store door (seat 7). Delete every public constructor/facade (`Db`, “open path → the database”) that returns one ambient handle granting storage + schema + evaluation as a single authority. A Catalog with its own persistence type is **Unconstructible**.
   - **E:** Catalog as a second durability organ grows a second door the moment implementation tires. “Third authority” means third **capability**, not third storage subsystem. Soft module split under one public `Db` is still reverse-composable. Proof: compile-fail `Store` evaluating a query; compile-fail `Engine` minting `Committed` without Store write capability; compile-fail any Catalog write/fsync/counter path; grep-delete of the fused constructor.
2. **WriteAuthority is an immutable affine signing capability; RecoveryMatrix advances epoch; IncarnationId separates nonce space**
   - **Q:** What affine capability alone authorizes the commit door — where do mutable floors live — what at-rest vs live-fork claim tags hold — and what optional quorum path recovers a lost token without zombie footprints?
   - **A:** `WriteAuthority` is an **immutable**, affine **signing** capability minted at genesis (or recovery/fork), held in the client keystore or HSM next to the root KEK, never in Store artifacts or leave-is-free packs, never `derive(KEK, …)`. It **authorizes the SweepDoor only**—no mutable watermark. Consume/reissue-through-fsync, token-held lease ceilings, and any second floor organ outside the WAL are **Unconstructible**. All mutable floors live in the Store WAL. Opening with matching `WriteAuthority` obtains session-only **`IncarnationMintCap`** and mints a durable **`IncarnationId`** (seat 62) before any lease. Opening without token → readers + `ForkGrant` materialize path—**unless** `RecoveryMatrix` applies. Address fence remains local mutex → `Refuse(StoreFenced)`. **Dual-use lineage:** two commit chains under the same CryptoDomain → poison; hostile keystore bit-copy and **live process/snapshot fork** that duplicate WriteAuthority use are **Unexposed until chains meet**—chain-meet protects **lineage integrity only**, never AEAD. **At-rest / clone honesty (seat 62):** historical-Id recycle within one WAL lineage is **Unconstructible** via `OpenOrdinal`; clone-vs-clone Id divergence rests on **Entropy** (cryptographically negligible under approved entropy arm)—never tag clone nonce distinctness Unconstructible. **Live-fork mid-session:** nonce/WA duplication is **Unexposed**—bounded per `StableCommitCap` arm (seat 27). HA is **token move**. Same-principal advance presents WriteAuthority into a new epoch session (new `IncarnationId`); different-principal successor mints a **new** WriteAuthority at discovery (seat 68). Capability lattice: **govern** (WriteAuthority / EpochGrant / ForkGrant / RecoveryGrant) ≠ **audit** (AuditKey) ≠ **read** (decrypt / AncestorReadGrant).
   - **RecoveryMatrix (optional, genesis-only).** M-of-N public keys sealed at creation; post-genesis mutation Unconstructible; recovery private keys are a **distinct custodian class**. Absent matrix → lost token means fork. Present: quorum `RecoveryGrant` (≠ EpochGrant ≠ WriteAuthority) = same-principal epoch advance: new FenceEpoch, CryptoDomain, WriteAuthority, **same KEK**, typed recovery root link; **in-place reissue Unconstructible**. **One-shot quorum:** signed payload includes predecessor `FenceEpoch` (+ GrantId / successor identity seed / key-material commitment—seat 68 seed law); `materialize(RecoveryGrant)` is deterministic and idempotent; a **second valid grant for the same predecessor epoch** is detectable quorum equivocation → poison for the signing set’s authority, **never a second lineage**. **Forward fencing (observed recovery):** a node that has **durably incorporated** the successor before admission and at SweepDoor commit **cannot** construct stale-epoch `Committed`—**Unconstructible at local admission and SweepDoor commit boundaries** (not distributed prevention); resurfaced old token is stale-epoch inert against that Store. Label: **observed-successor stale-epoch commit Unconstructible at local doors**. **Partition dual-history (oblivious holder):** a partitioned writer that never observed the grant may still mint a divergent chain under the old CryptoDomain—**Unexposed until chains meet**, then dual-chain poison + operator remediation (identical physics to ForkGrant’s oblivious original). Epoch advance provides forward local fencing, never prevention of offline dual-history. **IntentClear-exempt** under **epoch- and incarnation-indexed footprint law** (seat 36): prior-epoch / prior-incarnation footprints are not live locks; orphans that observed recovery land `Refuse(AuthorityRecovered)` at the next admission-visible write boundary. Successor NonceLease/`IncarnationId` space cannot collide with prior.
   - **E:** Detection cannot un-reuse AEAD; recycle Unconstructible is lineage-local; clone distinctness is Entropy; “stale-epoch Committed Unconstructible” for a partitioned oblivious holder is claim inflation. Recovery is revocation + epoch advance for observers, not proof of process death. Proof: two-clone at-rest DST (equal OpenOrdinal, differing Entropy); partitioned-old-writer-through-recovery DST → dual-chain poison + remediation surface; live-fork SIV arms; AuthorityRecovered DST; compile-fail watermark / token-floor.

4. **Store identity across rename, clone, fork, restore**
   - **Q:** Is Store identity location-addressed or content-addressed — when does clone preserve identity versus mint a fork — and which epoch-grant act advances versus forks?
   - **A:** Pick sealed **Store identity digest** (genesis / fork-minted id) — not path/URL, not tip state root. Path/URL is rebindable location only. Rename/move preserves Store identity. Restore of the same artifacts preserves **readable** identity; **write resume** requires the matching `WriteAuthority` (seat 2)—write continuity was never in the bytes. Copying artifacts without the token cannot construct resume-`Committed` on the same StoreId: the write path is `ForkGrant` materialization (new StoreId / CryptoDomain / WriteAuthority at **discovery**—seat 68) or RecoveryGrant. Intent is not a constructor; absence of `WriteAuthority` is. **Same-principal** `EpochGrant` successor advances `FenceEpoch` on the **same** StoreId under the held `WriteAuthority` (immutable; new session `IncarnationId`). **Different-principal** `ForkGrant` authorizes a successor identity; materialization is at **discovery** (seat 68)—grant does not reassign the original lineage’s post-grant bytes; original WriteAuthority stays with the original lineage. Equivalence *at a cut* uses state roots; fork-equivalence uses fork-point roots (seat 58).
   - **E:** Soft location-as-identity breaks capabilities. Soft intent-based clone left `cp -r` as multi-primary. Soft restore-implies-write-resume smuggles authority into leave-is-free. Proof: clone-without-token DST → reader or ForkGrant materialize only; resume DST requires token; same-principal advance keeps StoreId; different-principal ForkGrant materializes successor at discovery.
5. **Capability to open a Store**
   - **Q:** What capability types alone may open a Store for read versus write-resume — how are they minted — and what typed refuse makes path-only open Unconstructible?
   - **A:** Only a privately constructed `StoreOpen` capability may open a Store. It is minted by host/root grant over an explicit Store identity (seat 4), never from a filesystem path alone. Path-only open → `Refuse(MissingStoreOpenCapability)`; a successful path-only open is **Unconstructible**. Write-resume additionally requires presenting `WriteAuthority` (seat 2); open without it yields readers (and the fork-mint door), never same-lineage `Committed`. WASM / DO hosts hold `WriteAuthority` in platform keystores or refuse production `Committed` (density law).
   - **E:** Path-only open is ambient authority. Soft “any authenticated process may open any path” recreates a host-wide super-principal. Proof: private constructor; compile-fail `open(path)`; host presents StoreOpen + identity; write path requires WriteAuthority.
6. **Open capability is an unforgeable TAG value — graph-scoped, bitemporal grant**
   - **Q:** How does the Store-open capability ratchet in the Type Authority Graph — graph-scoped with no span across two Stores — and what ambient shared-secret or bind-address authority is deleted?

Delete shared-secret-as-authorization and bind-address/auth-skip as Store-open authority — transport may authenticate; it never authorizes open. `WriteAuthority` remains a distinct affine token (seat 2)—not substitutable by StoreOpen alone.
   - **E:** Multi-store sovereignty is graph-confinement. Secrets-as-authz and bind-skip recreate ambient super-principals. Proof: private TAG constructor; type forbids multi-Store scope; grep-delete secret-as-authz and bind-skip at open.
7. **Catalog is the schema interpreter; schema Records are Store facts**
   - **Q:** Are schema mutations ordinary KyzoRecords on the Store door — what does Catalog own — and what Catalog persistence path is Unconstructible?
   - **A:** Schema mutations are KyzoRecords, admitted through Engine, committed through the same Store WAL/`Committed` door, under the same epoch and root chain. Catalog is the Engine-held typed as-of interpreter over those Records—rebuildable from replay like every projection. Catalog gates admission of schema-mutating Records and evaluates schema as-of; it never persists.    - **E:** Ambiguity here is where the one-door law dies quietly. Soft Catalog-with-persistence recreates “the database.” Proof: schema put/retract through Store commit; compile-fail Catalog write path; as-of schema = as-of commit position; standing-view Catalog generation pin = Store commit pin.

### L4 — Durability
21. **WAL is sole commit-order authority in the retained suffix; memtables/SSTs are rebuildable cache**
    - **Q:** After wiping every memtable and SST, what single replay procedure reconstructs Engine-visible state — how does CheckpointSeal transfer authority for a truncated prefix — and how is commit order defined under SweepDoor group commit?
    - **A:** Within the **retained WAL suffix**, `WAL.replay()` from durable log segments alone reconstructs Engine-visible state; memtables/SSTs wipe and rebuild as cache. Truncation of a covered prefix is **Unconstructible** without a privately minted `CheckpointSeal` (seat 26). Logical commit order is **`CommitOrdinal`** assigned only inside the SweepDoor at the durable event (seat 25); predecessor-hash seals that ordinal. `IntentOrdinal` orders contenders and may gap; it is never sealed history. IntentionQueue arrival and SweepDoor batch composition are carriage—never order authority.
    - **E:** Soft “SST is source of truth after flush” fails wipe/replay. Soft truncate-without-seal makes recovery depend on a disposable dump. Soft arrival-order-as-authority makes batch composition schedule-dependent. Proof: wipe-SST/memtable test; truncate-without-seal compile-fail; chain positions define order inside a sweep fsync.
22. **Object durability before `Committed` meaning; ordinal StagingTTL; PermanenceCandidate; ObjectDurabilityClass; Repair**
    - **Q:** How does two-phase object naming stay honest — who decides expiry on busy vs idle Stores — what does Durable promise — what recoverable path exists after ObjectMissing — and what makes late confirm / fabric-clock expiry / immortal-orphan / non-identical Repair Unconstructible?
    - **A:** **Logical** `ObjectSlot` remains `Pending(StagingToken)` | `Durable(ObjectRef)`. **Physical** seam: `VolatilePending → PermanenceCandidate → Durable`. Staging seals `expires_at_transaction_cut` at the staging `Committed`: cut = stage `CommitOrdinal` + genesis-sealed **`StagingTTL` in dense CommitOrdinals** (post-genesis mutation **Unconstructible**). Fabric may execute deletion early or late vs wall clock; expiry authority is Store cut comparison only.
    - **Idle vs active (both ruled).** **Passive half (law):** expiry is relative to committed history; where no truth advances, nothing expires; an idle Store’s staged objects are **unresolved, not decayed**—by deliberate law, not a bug. Empty “tick commits” or maintenance timers that advance the cut without real truth are wall clock wearing a commit costume—**Unconstructible** as green Spec. **Active half:** Store **reclaim certificate** extends from PermanenceCandidate to **VolatilePending**: reclamation of any unconfirmed staged object is an explicit, governed, always-lawful act of the authority holder, idle or busy. Passive expiry is cut-decided; active reclamation is capability-decided; wall clock decides neither. Operator surface exposes **staged-object pressure** per Store.
    - **PermanenceCandidate** inherits `expires_at`. Before cut: confirm or reclaim; at or past cut: reclaim only. `PermanenceWitness::mint` takes current committed cut and is **Unconstructible** when `cut ≥ expires_at`. CheckpointSeal binds live candidate manifest (seat 26).
    - **`ObjectDurabilityClass` (product, not chain).** `PermanenceWitness` seals a product of guarantees declared per object-backend arm: at minimum `{ confirmed_copies, failure_domains, regions, consistency, integrity_verification, backend_contract }` with **sealed enum/newtype dimensions** (not open folklore). Each arm’s destructive campaign licenses the specific product it attests. Soft class-less Durable and soft total-order ladders (`SingleCopy ≤ ReplicatedN ≤ CrossRegion`) are deleted—those concealed operational incomparability.
    - **Dominance (partial order).** Class A dominates B iff every dimension of A is ≥ the corresponding dimension of B. **Repair:** re-stage bytes content-hash-verified against the sealed record; `PermanenceWitness::mint` requires the proposed class **dominates** the original—non-dominating Repair is **Unconstructible** at the constructor. Incomparable proposal → typed refuse carrying both classes; authority either supplies a dominating class or explicitly **`Downgrade`**—a named append-only supersession that records the class change as auditable fact, never silent. Hostile backend lying about attested class remains **Unexposed** (arm campaign / crate boundary).
    - **Client contract (normative).** Pending licenses the ref and the expiry decision—client handles `Decayed` and `ObjectMissing`. PermanenceCandidate licenses no ordinary read; may become Durable or reclaim-only. Durable licenses the sealed product class; client still handles corruption/unavailability.
    - Cut passed + resolve Pending → `Refuse(Decayed)`; fabric deleted while cut live → `Refuse(ObjectMissing)`. Confirm-then-strip Unconstructible. Fact commit and Durable supersession remain different `Committed` events.
